Legacy Java Testing Playbook with Claude

A practical operating model for using Claude to raise unit coverage, mutation coverage, and end-to-end test depth in a legacy Java application without over-automating the wrong parts.

Core recommendation: use Claude inside your own repo with deterministic scripts, test harnesses, and CI gates. Managed agents are usually not required for a single legacy Java application whose main goal is code understanding, test generation, execution, and iterative hardening.

1) What this playbook is optimizing for

Primary outcome

Increase meaningful test protection, not just vanity metrics.

Secondary outcome

Make legacy code more testable with small, safe refactors rather than large rewrites.

Constraint

Preserve production behavior unless a refactor is explicitly proposed and justified.

Reality check

100% unit, 100% mutation, and 100% e2e can be a governance target, but not always a good engineering target. This playbook shows how to pursue it with minimal waste.

2) Do you need managed agents?

Situation Best fit Why
One legacy Java repo, active developer involvement, frequent local runs Claude Code / IDE / API in your repo Fastest feedback loop. Easy to inspect diffs, run tests, and guide refactors.
Many repos, long-running autonomous task processing, queue-based orchestration Agent platform or managed agents Better when work must continue across sessions and systems.
Strict compliance, internal infrastructure, custom toolchain Self-hosted orchestration around Claude API Lets you enforce your own controls, audit, and execution model.
Default answer: no, you do not need managed agents for this testing problem. You need a disciplined repo workflow, clear instructions, and CI that enforces the non-negotiables.

3) Best operating model

  1. Claude explores the codebase and produces a testability map.
  2. Claude adds characterization tests before risky refactors.
  3. Claude proposes tiny refactors that create seams for dependency injection and observability.
  4. Unit tests are generated in small batches and validated by real commands.
  5. PIT mutation results are used to harden weak tests.
  6. E2E is focused on critical business journeys, not every line of code.
  7. CI gates decide pass/fail. The model does not.

4) Recommended test stack

Unit tests

JUnit 5, Mockito, AssertJ

Coverage

JaCoCo for line and branch coverage

Mutation

PIT (pitest) for mutation score and surviving mutant analysis

Integration boundaries

Testcontainers, WireMock, MockWebServer, REST-assured

E2E UI

Playwright if you have a browser UI

Execution rails

Makefile, Gradle tasks, Maven profiles, CI pipelines, pre-merge gates

5) Coverage philosophy for legacy systems

Metric What to optimize Common trap
Unit coverage Behavior and branch coverage in business logic Trivial tests that only instantiate objects or call getters
Mutation coverage Strong assertions and observable outcomes Adding more mocks instead of better assertions
E2E coverage Critical user journeys and failure recovery Trying to cover every branch through browser tests

6) Rollout phases

Phase 1 — Create a testability map

  • Inventory modules, frameworks, and package responsibilities.
  • Identify side-effect boundaries: DB, HTTP, file system, queues, scheduler, time, randomness.
  • Flag code smells: God classes, static global access, hidden singletons, large methods, mixed concerns.
  • Rank packages by business criticality and change frequency.
Expected deliverable from Claude:
- package/module inventory
- risk map
- test seam candidates
- likely dead code
- easiest coverage wins
- highest-value mutation targets
- highest-value e2e flows

Phase 2 — Freeze behavior with characterization tests

  • Write tests around existing behavior before refactoring fragile code.
  • Capture current output, persistence effects, emitted events, and error handling.
  • Use approval/golden style outputs only when behavior is hard to express with fine-grained assertions.

Phase 3 — Introduce test seams

  • Wrap static calls and globals behind interfaces.
  • Extract pure logic from orchestration-heavy classes.
  • Inject clocks, UUID generators, HTTP clients, repositories, and adapters.
  • Break giant methods into observable units.

Phase 4 — Raise unit and branch coverage

  • Generate tests package by package, not repo-wide all at once.
  • Require every test to state the behavior it protects.
  • Prefer AssertJ-rich assertions and state verification over mocking everything.

Phase 5 — Mutation hardening

  • Run PIT on touched modules first.
  • For every surviving mutant, classify it:
    • missing assertion
    • too much mocking
    • dead code
    • equivalent mutant
    • needs small refactor for observability
  • Strengthen tests before adding new ones.

Phase 6 — Lean E2E for critical flows

  • Cover the smallest set of high-value workflows that proves the system works as a whole.
  • Include failure and recovery, not just happy paths.
  • Push combinatorial detail down into unit/integration tests.

7) Practical definition of “100%”

Recommended interpretation: if leadership requires 100%, define it precisely. “100% e2e coverage” should mean all critical business flows and key states are covered, not that every line is hit through UI automation.
Area Reasonable interpretation
100% unit All production classes in scope have line and branch tests, with explicit exclusions for generated/framework glue if policy allows.
100% mutation All killable mutants in scope are killed, with reviewed suppressions for true equivalent mutants.
100% e2e All mission-critical workflows, roles, major states, and failure-recovery paths are covered by system-level tests.

8) Repo conventions for Claude

Create a CLAUDE.md file at the repo root with strict rules.

# CLAUDE.md

You are acting as a senior Java test architect for this legacy application.

Goals:
1) maximize meaningful unit and branch coverage
2) maximize meaningful mutation coverage with PIT
3) create a lean but complete e2e suite for critical business journeys
4) preserve production behavior unless a refactor is explicitly proposed and justified

Rules:
- Never change production logic just to satisfy tests.
- Start with characterization tests for fragile legacy areas.
- Prefer stronger assertions over more mocks.
- Prefer integration tests when behavior crosses boundaries and heavy mocking would hide risk.
- For every generated test, add a short comment describing the behavior under test.
- When mutation survives, first strengthen assertions, then consider a small refactor for observability.
- Keep diffs small and grouped by package or workflow.
- After changes, run the repo's mandated commands and report the results.

Required report after each batch:
- files changed
- tests added
- coverage delta
- mutation delta
- surviving mutants and likely reason
- risky assumptions or refactors

9) Command rails Claude should always run

Wrap your project commands so Claude can execute the same standard path every time.

# Example Makefile
unit:
	./mvnw -q -DskipITs test

coverage:
	./mvnw -q test jacoco:report

mutation:
	./mvnw -q -DskipITs org.pitest:pitest-maven:mutationCoverage

integration:
	./mvnw -q -Pintegration verify

e2e:
	./run-e2e.sh

smoke:
	make unit coverage

full-check:
	make unit coverage mutation integration e2e

JaCoCo

<plugin>
  <groupId>org.jacoco</groupId>
  <artifactId>jacoco-maven-plugin</artifactId>
  <version>0.8.11</version>
  <executions>
    <execution>
      <goals>
        <goal>prepare-agent</goal>
      </goals>
    </execution>
    <execution>
      <id>report</id>
      <phase>test</phase>
      <goals>
        <goal>report</goal>
        <goal>check</goal>
      </goals>
      <configuration>
        <rules>
          <rule>
            <element>BUNDLE</element>
            <limits>
              <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>1.00</minimum>
              </limit>
              <limit>
                <counter>BRANCH</counter>
                <value>COVEREDRATIO</value>
                <minimum>1.00</minimum>
              </limit>
            </limits>
          </rule>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>

PIT

<plugin>
  <groupId>org.pitest</groupId>
  <artifactId>pitest-maven</artifactId>
  <version>1.15.8</version>
  <configuration>
    <targetClasses>
      <param>com.yourcompany.*</param>
    </targetClasses>
    <targetTests>
      <param>com.yourcompany.*</param>
    </targetTests>
    <mutationThreshold>100</mutationThreshold>
    <outputFormats>
      <param>HTML</param>
      <param>XML</param>
    </outputFormats>
    <threads>4</threads>
    <timestampedReports>false</timestampedReports>
  </configuration>
</plugin>
Start with lower thresholds on the first pass if your repo is far from the target, then ratchet upward per package. If leadership insists on immediate 100%, expect significant cleanup and suppression review work.

13) How Claude should work package by package

Iteration loop:
1. Pick one package or one business workflow.
2. Read production code and existing tests.
3. Summarize current behavior and risks.
4. Add or improve characterization tests.
5. Propose the smallest safe refactor if testability is poor.
6. Generate unit tests.
7. Run unit + coverage.
8. Run PIT on touched scope.
9. Analyze surviving mutants.
10. Strengthen tests or refactor for observability.
11. Run smoke e2e if the workflow is user-visible.
12. Produce a concise report.

14) Claude prompts you can reuse

A. Initial mapping prompt

You are reviewing a legacy Java application to improve meaningful unit coverage, mutation coverage, and e2e reliability.

Tasks:
- map the package structure and module responsibilities
- identify business-critical flows
- identify side-effect boundaries: DB, HTTP, file system, queues, schedulers, time, randomness
- identify hard-to-test areas and why they are hard to test
- identify dead code or suspicious low-value code
- recommend a phased coverage plan ranked by value and difficulty

Output format:
1. package/module map
2. critical workflow map
3. test seam opportunities
4. top 10 coverage wins
5. top 10 mutation-risk areas
6. top 10 e2e workflow candidates

B. Unit test generation prompt

Act as a senior Java test engineer.

For the selected package:
- read existing production code and tests
- do not change production behavior
- generate tests for meaningful branches and edge cases
- avoid trivial tests and avoid overspecifying internals
- prefer assertive behavior-focused tests over mock-heavy tests
- comment each test with the behavior it protects
- run tests and report failures clearly

At the end provide:
- files changed
- why each new test matters
- missing seams that limit test quality

C. Mutation analysis prompt

PIT reports surviving mutants in the selected package.

For each survivor:
- explain what behavior changed
- classify the reason the mutant survived
- propose the smallest improvement to kill it
- prefer stronger assertions before adding mocks
- identify equivalent mutants separately
- only propose production refactors if they improve observability or separability without changing behavior

Return a table with:
- class
- mutant type
- likely reason survived
- recommended fix
- whether it is likely equivalent

D. E2E design prompt

Design a lean e2e suite for this application.

Requirements:
- cover business-critical workflows only
- include auth/permissions, validation failure, dependency failure, retry/recovery, and persistence verification where relevant
- minimize brittle UI selectors and repeated setup
- identify which checks belong in e2e versus integration versus unit

Output:
1. prioritized flow list
2. reasons each flow belongs in e2e
3. Playwright or REST-assured test skeletons
4. test data and environment requirements

15) How to handle surviving mutants intelligently

Mutant pattern What it usually means What to do
Conditional boundary changed and survived Edge case missing Add boundary assertions for exact branch behavior
Return value mutation survived Test never asserted the actual outcome Assert returned data, persisted state, or emitted side effect
Void call removal survived Interaction or side-effect not observed Verify side effect through state or meaningful interaction checks
Math/operator mutation survived Weak assertions Strengthen exact-value assertions and boundary cases
Equivalent mutant No observable semantic difference Suppress with review, document why

16) E2E flow model

For a legacy business app, define e2e around workflows, not pages.

Recommended e2e flow categories:
- sign in / role-based access
- create primary business entity
- edit and persist changes
- invalid input / validation handling
- approval or review workflow
- downstream dependency failure and user-visible recovery
- retries / idempotency / duplicate submission protection
- search / retrieval / reporting path
- audit trail or event emission verification
- logout / session expiration / permission denial

17) Anti-patterns to avoid

  • Changing production logic only to make tests pass.
  • Generating thousands of shallow tests with almost no assertions.
  • Mocking the entire world so mutation survives anyway.
  • Trying to get 100% e2e by pushing all branch coverage into browser tests.
  • Running PIT across the entire monolith on every tiny change.
  • Ignoring equivalent mutants and wasting time fighting the tool.

18) Suggested implementation path for the first 30 days

Week Primary goal Output
1 Map the repo, define scope, add command rails, baseline metrics Coverage baseline, mutation baseline, flow inventory, CLAUDE.md, CI skeleton
2 Characterization tests and seams for top-risk packages Stable first package, safer refactor boundaries
3 Raise unit and branch coverage package by package Visible progress on critical modules
4 PIT hardening and first lean e2e pack Higher confidence tests, smoke system validation

19) Final recommendation

The best way to use Claude here is: Claude in-repo + strict repo instructions + deterministic commands + CI gates + phased coverage strategy.

Use managed agents only if you need cloud-hosted, stateful, cross-session autonomous execution at scale. For a legacy Java testing push, they are usually optional, not necessary.
