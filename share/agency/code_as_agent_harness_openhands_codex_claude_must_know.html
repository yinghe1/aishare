1. One-line idea

Code as Agent Harness: code is no longer only the output of an AI model; code becomes the runtime layer that lets an agent act, observe, verify, remember, and improve.

The paper argues that the important unit is not just the LLM. The important unit is the LLM plus harness: tools, files, shell, APIs, sandbox, tests, logs, memory, approvals, and audit trail.

2. Mental model shift

Old model:

User asks → LLM writes code → code is final answer

New agent model:

User goal
  ↓
Agent plans
  ↓
Agent edits files / runs commands / calls tools
  ↓
Harness executes in a controlled runtime
  ↓
Tests, logs, errors, policies, and human feedback return as signals
  ↓
Agent revises, persists state, and produces an artifact

Must know: agent quality is not only model quality. It depends on the harness: tool design, sandbox isolation, state management, verification, and governance.

3. The paper framework

LayerMeaningMust know
Harness interfaceHow code connects the model to reasoning, action, and environment state.Files, shell, APIs, tests, browser, repo state, logs.
Harness mechanismsThe control surfaces around the model.Planning, memory, context construction, tool use, feedback control, verification.
Scaling layerHow one agent becomes many coordinated agents.Planner, coder, tester, reviewer, security checker using shared executable state.

The paper’s practical message: do not treat agents as chatbots with tools. Treat them as controlled execution systems.

4. OpenHands must know

OpenHands is a useful concrete example because it exposes much of the harness as inspectable software: SDK, CLI, web app, sandbox, tools, workspace, agent server, and deployment options.

1. SDK first

The Software Agent SDK provides Python and REST APIs for building agents that work with code.

2. Interfaces are separate

CLI, web, cloud, and integrations consume the SDK instead of becoming the agent core.

3. Sandbox is the action zone

OpenHands supports Docker, process, and remote sandbox providers. Docker is recommended for isolation.

4. Tools define capability

File edit, Bash, browsing, tests, MCP/custom tools, and repo operations form the agent’s action space.

5. Events create inspectability

Useful output is not only the final diff; it is also the run history: observations, actions, failures, tests, and decisions.

6. Model-agnostic by design

The harness should survive model changes; tools, policies, and verifiers should not assume one model’s behavior.

Practical interpretation: OpenHands is not merely a terminal coding assistant. It is closer to an open-source platform for building, inspecting, customizing, and deploying coding agents.

5. OpenHands vs Codex CLI vs Claude Code

All three can feel similar at the terminal: they read a repo, edit files, run commands, and iterate from test output. The difference is mainly product scope and extensibility, not the surface chat loop.

Shortest distinction: Claude Code is a polished coding-agent product. Codex CLI is a focused local coding-agent tool. OpenHands is the broader open-source harness platform.

6. Reference architecture

User / Team Goal │ ▼ Interface CLI · Web · IDE · API · GitHub/GitLab/Slack/Ticketing │ ▼ Agent Core / SDK LLM adapter · agent loop · planning · conversation state · events │ ▼ Tool Layer File edit · Bash · browser · MCP · custom tools · test runners │ ▼ Workspace / Sandbox Docker · process · remote sandbox · filesystem · services │ ▼ Verification / Governance Tests · lint · security checks · policies · approvals · audit trail │ ▼ Artifacts Diff · PR · docs · logs · test output · decisions · reusable skills

This is the real “code as agent harness” unit: model + runtime + tools + state + verifier + governance.

7. Evaluation and governance must know

QuestionWhy it matters
Did it solve the task?Final success still matters.
Did it prove the task?Weak tests create fake success.
Was the trajectory safe?Agents can run commands and change files; policy and sandbox boundaries matter.
Can the run be replayed?Auditability is essential for enterprise trust.
Did it avoid context pollution?Bad memory or irrelevant context makes long tasks worse.
Can humans intervene at the right points?Approval should become durable governance state, not just a one-time pause.

Biggest trap: a coding agent can look successful because it produced a plausible patch. The real question is whether the harness had a strong verifier.

8. Concise learning path

  1. PaperLearn the vocabulary: interface, mechanisms, feedback, verification, multi-agent scaling.
  2. OpenHands CLIRun a small repo task and watch file reads, edits, commands, failures, and recovery.
  3. SandboxUnderstand Docker vs process vs remote sandbox tradeoffs.
  4. SDKMap agents, tools, conversations, events, workspace, and security policies.
  5. Custom toolAdd one tool such as a repo scanner, compliance checker, or test summarizer.
  6. VerifierAdd required tests, lint, security checks, and human approval gates.
  7. CompareUse Codex CLI and Claude Code as product references; use OpenHands to study and modify the platform layer.
