The Journey of a Single Task

What OpenHands actually is, and what happens between “build me a feature” and a pull request appearing on GitHub.

What you're looking at

OpenHands is a platform for running AI software-engineering agents — programs that read a codebase, write code, run commands, and open pull requests on your behalf. It scores 77.6 on SWE-bench, near the top of the field.

But “OpenHands” is really five products sharing one engine. This course is about the repository in front of you — the server and web application that turns that engine into a multi-user product.

📦

The SDK

A Python library — the actual agent engine. Lives in a separate repo and ships here as the openhands-agent-server package.

⌨️

The CLI

A terminal tool like Claude Code or Codex. Drives the same engine from your shell.

🖥️

The Local GUI

A REST API + React app for running agents on your laptop. This repo.

☁️

Cloud + Enterprise

The same GUI, scaled to many users with auth, billing, and integrations. The enterprise/ directory here.

🧭
Orient yourself before you read code

This repository is the application layer: openhands/app_server/ (the REST API), frontend/ (the React app), and enterprise/ (the multi-tenant SaaS). The agent's reasoning loop lives in the SDK, a dependency. Keep that boundary in mind — it explains almost every design choice you'll meet.

The metaphor: mission control and the spacecraft

Picture a space mission. There's mission control on the ground — it doesn't fly anywhere, but it decides when to launch, watches telemetry, and keeps records. And there's the spacecraft — isolated, doing the actual dangerous work out in the void.

OpenHands is built exactly this way. The app server is mission control. The agent, running inside a locked-down sandbox, is the spacecraft. They are two separate processes that talk over the network.

🛰️
Mission control = the App Server

A FastAPI service. Starts conversations, launches sandboxes, stores events, enforces auth. Never runs the agent's code itself.

🚀
The spacecraft = the Agent Server

Runs inside the sandbox. This is where the LLM loop, tool calls, and shell commands actually execute. Exposed to mission control over an HTTP port.

02

The Cast: Control Plane vs. Data Plane

Meet the services that make up mission control — and the hard line that separates them from the agent.

An orchestra, not a soloist

An orchestra has no single “music maker.” A conductor coordinates; strings, brass, and percussion each own one job. Remove the conductor and you get noise; remove the brass and the piece is thinner but still plays.

The app server is an orchestra of focused services. Each owns one responsibility and knows nothing about the others' internals. The separation of concerns is deliberate and total.

🎼
AppConversationService

The conductor. Creates a conversation, wires up the LLM, condenser, security policy, and skills, then drives the start sequence.

📦
SandboxService

Launches, health-checks, pauses, resumes, and reaps the isolated environments the agent runs in.

📜
EventService

The scribe. Persists and streams every event a conversation produces, backed by the filesystem, S3, or Google Cloud.

🔔
EventCallbackService

The reactor. Runs custom processors when matching events arrive — the extension point you'll use in Module 6.

📈
AnalyticsService & UserContext

Who is this user, what may they do, and what should we measure — consent-gated.

The line down the middle: two planes

Cloud architecture has a name for this split. The control plane makes decisions and holds state; the data plane does the actual work on the actual data. OpenHands draws that line between the app server and the agent.

🛰️

Control Plane · App Server

FastAPI. Auth, conversation records, sandbox orchestration, event storage, billing. Multi-tenant. Trusted. Never executes agent-authored code.

🚀

Data Plane · Agent Server

Runs in the sandbox. The LLM loop, file edits, shell commands, browser. Single-tenant per conversation. Untrusted. Disposable.

🏛️
Design consideration: why split the planes at all?

Three reasons, in priority order. (1) Security — agent-authored shell commands must never share a process with multi-tenant user data. (2) Scaling — one control plane can fan out to thousands of independent agent sandboxes. (3) Licensing — the core openhands and agent-server images are MIT, while enterprise/ is source-available; a clean process boundary keeps that line enforceable. The cost you pay: every interaction is now a network round-trip with its own failure modes.

How the planes actually touch

The control plane reaches the data plane through one narrow channel: an HTTP URL the sandbox exposes, plus a session key. Here's the real lookup that finds it, from sandbox_service.py:

CODE 
def _get_agent_server_url(self, sandbox: SandboxInfo) -> str:
    if not sandbox.exposed_urls:
        raise SandboxError(f'No exposed URLs for sandbox: {sandbox.id}')
    for exposed_url in sandbox.exposed_urls:
        if exposed_url.name == AGENT_SERVER:
            return replace_localhost_hostname_for_docker(exposed_url.url)
    raise SandboxError(f'No agent server URL found for sandbox: {sandbox.id}')
PLAIN ENGLISH

Find the address mission control should call for this sandbox.

If the sandbox published no URLs yet, that's an error — there's nothing to talk to.

Otherwise, scan the URLs the sandbox exposed…

…looking for the one named “agent server.”

Found it — but rewrite localhost so it works from inside Docker's network, then hand it back.

No agent-server URL? Fail loudly rather than guess.

🔌
Notice the discipline

The control plane holds no direct reference to the agent process — only a URL and a key it discovered at runtime. That indirection is what lets the same code drive a local Docker container, a remote runtime, or a Kubernetes pod without changing.

03

The Injector Pattern

How every service in OpenHands becomes swappable — and why the architects chose this over hardcoding.

The standardized wall socket

A lamp doesn't care which power plant it's connected to — coal, solar, nuclear. It plugs into a standardized socket, and the wiring behind the wall is someone else's problem. Swap the plant; the lamp never knows.

OpenHands wires every service this way. Code depends on an abstract socket (an interface), and a small “injector” decides at startup which real implementation gets plugged in. This is dependency injection, and it is the single most important pattern in this repository.

🧩
The repeating shape

Almost every service in app_server follows the same three-part shape: an abstract base class (the socket), one or more concrete implementations (the appliances), and an Injector (the wiring that picks one). Learn it once and the whole codebase opens up.

The socket itself

Here is the entire Injector base class. It's tiny on purpose — its job is just to yield a ready-to-use object and clean it up afterward:

CODE 
class Injector(Generic[T], ABC):
    """Object designed to facilitate dependency injection"""

    async def depends(self, request: Request) -> AsyncGenerator[T, None]:
        """Depends function suitable for use with FastAPI DI."""
        async for result in self.inject(request.state, request):
            yield result
PLAIN ENGLISH

A generic socket that knows how to produce some type T.

 

 

This method plugs the socket straight into FastAPI's own dependency system…

 

…so any route can just say “give me the service” and get the right one, fully wired, for this request.

Each concrete service then declares its socket. For event callbacks it's one line:

CODE 
class EventCallbackServiceInjector(
    DiscriminatedUnionMixin, Injector[EventCallbackService], ABC
):
    pass
PLAIN ENGLISH

A socket that specifically produces an EventCallbackService…

…and mixes in DiscriminatedUnion — the magic that lets the choice of implementation be written down in config and saved to disk.

 

No body needed — the shape is the contract.

Why DiscriminatedUnion matters

A plain interface lets you swap implementations in code. A discriminated union lets you swap them in data — a JSON object or an environment variable can name the class and carry its settings. That single property is what makes the whole config-driven architecture possible.

🗂️

FilesystemEventService

The default. Stores events as files on local disk. Perfect for a laptop.

🪣

AwsEventService

Same socket, S3 behind it. Chosen when the storage provider is AWS.

☁️

GoogleCloudEventService

Same socket again, GCS behind it. The conductor never notices the difference.

Where the choice is made

One function reads the environment and fills in any socket the operator didn't override. This is the actual wiring for event storage in config_from_env():

CODE 
config: AppServerConfig = from_env(AppServerConfig, 'OH')

if config.event is None:
    provider = get_storage_provider()
    if provider == StorageProvider.AWS:
        config.event = AwsEventServiceInjector(bucket_name=bucket_name)
    elif provider == StorageProvider.GCP:
        config.event = GoogleCloudEventServiceInjector(bucket_name=bucket_name)
    else:
        config.event = FilesystemEventServiceInjector()
PLAIN ENGLISH

Build the whole config from OH_* environment variables in one shot.

 

If the operator didn't explicitly pin an event backend…

…look at which storage provider is configured…

…AWS? plug in the S3 socket…

 

…GCP? plug in the Google Cloud socket…

 

…otherwise fall back to plain files. Sensible default, full override.

🎚️
“Convention with an escape hatch”

Notice the pattern: if config.X is None: pick a smart default. Operators get zero-config on a laptop, yet can override any single service by setting OH_* variables — without forking the code. This is the design philosophy of the entire config layer.

The honest trade-off

This pattern is not free. An architect should weigh both sides before adopting it.

✅

What you gain

One codebase runs laptop → cloud → enterprise. Tests inject fakes trivially. New backends are additive, not invasive. Config is data, so it's serializable and auditable.

⚠️

What it costs

Indirection: to read “what actually runs,” you trace base → injector → config_from_env. The from_env magic is implicit. Newcomers must learn the shape before any single feature makes sense.

🔍
Debugging tip you'll actually use

When something behaves unexpectedly, “what's plugged in right now?” is the first question. The answer is never in the calling code — it's in config_from_env() and the OH_* environment. Start there, not at the call site.

The state machine, in code

This is the real run_setup_scripts method. The interesting part is the yield after each status change — that's what powers the live progress you see in the UI:

CODE 
task.status = AppConversationStartTaskStatus.PREPARING_REPOSITORY
yield task

task.status = AppConversationStartTaskStatus.RUNNING_SETUP_SCRIPT

PLAIN ENGLISH

Mark the task as “preparing the repository.”

Then actually do the slow thing: clone or initialise the repo.

 

Now we're “running the setup script.” Announce it…

 

…then run .openhands/setup.sh.

 

Finally, “setting up git hooks.” Announce, then install.

📡
Design consideration: announce, then act

Using a generator that yields before each slow step is a deliberate UX-and-architecture decision: the user sees “cloning…” the instant it begins, not after it finishes. The status is the contract between a long backend job and a responsive frontend.

The sandbox lifecycle: more than start

The SandboxService abstract base defines a full lifecycle — not just “start.” Each is one socket on the wall (Module 3), implemented by Docker, a local process, or a remote runtime.

start_sandboxProvision a new isolated environment for a conversation.
wait_for_sandbox_runningPoll until status is RUNNING and the agent server answers /alive.
pause_sandbox / resume_sandboxFreeze an idle sandbox to reclaim resources; thaw it on demand.
pause_old_sandboxesCap concurrent sandboxes per user by freezing the oldest ones.
delete_sandboxStop and tear down — the disposable nature made literal.

A race condition the code guards against

A sandbox reporting RUNNING doesn't mean the agent inside is ready to take requests. The base class handles that gap explicitly:

CODE 
if sandbox.status == SandboxStatus.RUNNING:
    # sandbox says RUNNING, but is the agent server actually up?
    if httpx_client and sandbox.exposed_urls:
        if await self._check_agent_server_alive(sandbox, httpx_client):
            return sandbox
        # not ready yet, keep polling
    else:
        return sandbox
PLAIN ENGLISH

The container reports it's running…

…but “the box is on” isn't “the worker is ready.”

If we were given a way to check and the sandbox has exposed a URL…

…actually knock on the agent's /alive door.

It answered — now it's safe to hand back.

Door's silent? Don't return yet; loop and try again.

 

(If no checker was supplied, trust the status and move on.)

⏱️
A real-world failure this prevents

Without the /alive check, the app server would fire the agent's first request into a process that isn't listening yet — an intermittent “connection refused” that only shows up under load or on slow machines. Health-gating the readiness check turns a flaky race into a deterministic wait.

05

Events & Callbacks: The Extension Spine

Everything the agent does becomes an event — and events are the seam where you bolt on your own behavior.

A house full of automation rules

In a smart home, you don't rewire the house to add a behavior. You write a rule: “when the front door opens, turn on the hallway light and text me.” The door knows nothing about lights or phones — it just emits an event, and independent rules react.

OpenHands works identically. The agent emits events — a message, a tool call, a finished run — and registered callbacks react. The agent never knows who's listening. This is publish/subscribe, and it's the cleanest place to extend the platform.

📜
Two halves: storage and reaction

The EventService is durable memory — it persists and streams the full event history (to disk, S3, or GCS, chosen by injector). The EventCallbackService is live reaction — it runs your code when a matching event arrives. Module 6 lives entirely in that second half.

The contract you implement: one method

To react to events, you subclass EventCallbackProcessor and implement a single async method. Here's the entire base class — this is the whole extension API:

CODE 
class EventCallbackProcessor(DiscriminatedUnionMixin, ABC):
    event_kind: ClassVar[EventKind] = 'MessageEvent'

    @abstractmethod
    async def __call__(
        self,
        conversation_id: UUID,
        callback: EventCallback,
        event: Event,
    ) -> EventCallbackResult | None:
        """Process an event."""
PLAIN ENGLISH

Define a reaction. (It's a discriminated union, so — like injectors — it can be stored as config and rebuilt.)

Declare which kind of event you care about. Default: chat messages.

 

You must fill in one thing…

…a callable that receives:

 

— which conversation this is,

— the callback's own record (so it can update itself),

— and the event that just fired.

Return a result to record, or None to stay silent.

🪝
This tiny interface is the whole point of the module

Logging, metrics, tracing, evaluation, webhooks, auto-naming a conversation — all of them are just different bodies for this one __call__ method. The platform ships LoggingCallbackProcessor as a copy-paste starting point and a title-setter as a real production example.

Resilience by construction

The dispatcher's error handling is worth reading carefully — it's a deliberate reliability decision. This is the real execute_callback:

CODE 
try:
    result = await callback.processor(conversation_id, callback, event)
    if result is None:
        return
    stored_result = StoredEventCallbackResult(**result.model_dump())
except Exception as exc:
    _logger.exception(f'Exception in callback {callback.id}')
    stored_result = StoredEventCallbackResult(
        status=EventCallbackResultStatus.ERROR, detail=str(exc), ...)
PLAIN ENGLISH

Run the user's processor…

 

If it chose to stay silent, we're done.

 

Otherwise, save what it returned.

But if it blew up…

…log the full stack trace for debugging…

…and record an ERROR result with the message — instead of letting the crash escape.

 

🛡️
A double-edged decision — note it for Module 7

Swallowing exceptions means your buggy observability hook can never take down a user's agent run. That's the right default. The cost: a silently failing callback leaves only an ERROR row in a table no one is watching. Resilience here trades loud failure for quiet failure — a trade we'll revisit honestly.

Customizing Observability & Evaluation

The applied module: wire your own tracing and your own trajectory scoring onto the event spine — with an honest map of what ships vs. what you build.

A reading convention for this module

Because we're about to mix real platform code with code you would write, every code block here is labelled. Don't skip these — they're the difference between “ships in the box” and “you implement it.”

The metaphor: a flight data recorder and a panel of judges

Observability is the black box — it records what happened so you can replay any flight. Evaluation is the panel of judges — it watches the recording and assigns a score. OpenHands gives you mounting points for both; the recorder is largely built, the judges are mostly yours to write.

Observability, layer 1: the analytics that ships

The platform already records product-level signals through AnalyticsService — a thin, consent-gated wrapper over PostHog. Two design choices in this REAL snippet are worth copying in your own telemetry:

REAL · analytics_service.py 
def capture(self, ctx, event, properties=None, session_id=None):
    if not ctx.consented:
        return
    merged = self._common_properties(org_id=ctx.org_id, session_id=session_id)
    if properties:
        merged.update(properties)
    try:
        self._client.capture(distinct_id=..., event=event, properties=merged)
    except Exception:
        logger.exception('AnalyticsService.capture failed for event=%s', event)
PLAIN ENGLISH

Record a server-side event…

Choice 1 — consent gate: if the user hasn't opted in,

do absolutely nothing. Privacy is the default, not an afterthought.

Attach common context (org, session) to every event…

…then merge in the caller's specifics.

 

Send it…

Choice 2 — never let telemetry break the app:

if the analytics SDK throws, log and swallow. Measuring must never crash the thing being measured.

📊
It already tracks the metrics evaluation cares about

track_conversation_finished (also REAL) emits terminal_state, turn_count, accumulated_cost_usd, and token counts. Those four fields are the raw material of agent evaluation — hold that thought for the second half.

PLAIN ENGLISH

Give mission control your Laminar credentials…

…the project key…

…and where to send traces.

 

Because the prefix is on the auto-forward list…

…the spacecraft inherits them and starts emitting traces. Config, not code.

🧭
Honest limitation

This binds you to Laminar specifically. There is no vendor-neutral OpenTelemetry exporter wired into the app server today. Want Datadog or Honeycomb instead? You either point Laminar at an OTel collector downstream, or you build your own hook — which is exactly layer 3.

Observability, layer 3: your own tracing hook

For control-plane-side observability (or a non-Laminar backend), implement an EventCallbackProcessor. This SAMPLE turns every event into a span on a backend of your choice — note it implements the exact REAL contract from Module 5:

SAMPLE · you write this 
class TracingCallbackProcessor(EventCallbackProcessor):
    event_kind: ClassVar[EventKind] = 'MessageEvent'

    async def __call__(self, conversation_id, callback, event):
        with tracer.start_as_current_span('agent.event') as span:
            span.set_attribute('conversation_id', str(conversation_id))
            span.set_attribute('event.kind', event.kind)
        return EventCallbackResult(
            status=EventCallbackResultStatus.SUCCESS,
            event_callback_id=callback.id, event_id=event.id,
            conversation_id=conversation_id)
PLAIN ENGLISH

Define a reaction — same base class the platform's own callbacks use.

Tell the dispatcher which events to send you.

 

Each time one arrives…

…open a span on your tracing backend (OTel, Langfuse, anything)…

…tag it with the conversation…

…and the event type.

Then hand back a SUCCESS result…

 

…so the dispatcher records that your hook ran cleanly. That's the entire integration.

🔁
Why this is the same shape as LoggingCallbackProcessor

The repo's REAL LoggingCallbackProcessor does precisely this, but its body just calls _logger.info(...). Swap the logger for a tracer and you have layer-3 observability. The platform handed you a working template; you change four lines.

Evaluation: the honest map first

Before any sample, the truth about where evaluation lives:

🚫

Not in this repo

The benchmark harness behind the 77.6 SWE-bench score lives in a separate project (OpenHands/benchmarks). There is no first-class “evaluation framework” injector in app_server today.

✅

What this repo gives you

The raw material: the full event stream, terminal state, turn count, and cost. Enough to score a trajectory yourself, on the same callback spine.

⚖️
Why honesty matters here

If someone tells you OpenHands has “built-in custom evaluation,” push back. It has the data and the hook for evaluation. The scoring logic — and any benchmark dataset, sandboxed test runner, or LLM-judge — is something you assemble. Knowing that boundary prevents a planning mistake.

Evaluation: a sample trajectory scorer

Here's how you'd score a finished run in-process. This SAMPLE reacts to the agent-completion event and records a score as a callback result — queryable later, no separate pipeline:

SAMPLE · you write this 
class TrajectoryScoringProcessor(EventCallbackProcessor):
    # set to the SDK event kind that marks completion
    event_kind: ClassVar[EventKind] = 'AgentStateChangedEvent'

    async def __call__(self, conversation_id, callback, event):
        score = await self.judge(conversation_id)   # tests / LLM-judge
        return EventCallbackResult(
            status=EventCallbackResultStatus.SUCCESS,
            detail=json.dumps({'score': score}),
            event_callback_id=callback.id, event_id=event.id,
            conversation_id=conversation_id)
PLAIN ENGLISH

A reaction — again, the same Module 5 contract.

Listen for the event that means “the agent reached a terminal state.”

(The exact name comes from the SDK's event taxonomy — verify it; don't trust this string blindly.)

 

When the run ends…

…judge it — run the repo's tests, or ask an LLM to grade the trajectory.

Record the verdict…

 

…as structured JSON in the result…

 

…tied to this exact conversation, persisted automatically.

🎯
Honest limitations of in-process scoring

(1) Judging is genuinely hard — “did the agent do a good job?” rarely has a clean signal; LLM-judges are noisy and can be gamed. (2) Callbacks aren't a batch harness — for reproducible benchmarks across hundreds of tasks, the external benchmarks repo is the right tool, not per-conversation callbacks. (3) The completion event kind must be verified against the SDK version you run — event taxonomies change with version bumps.

07

Trade-offs, Honest Limits & The Road Ahead

The architect's wrap-up: why the big decisions were made, where they hurt, and a clearly-labelled look at what could come next.

The five decisions, and what each one cost

Every architecture is a set of bets. Here are the five that define this codebase — each with the reasoning and the bill.

🛰️

Two-plane split

Why: security isolation, horizontal scale, MIT/enterprise licensing boundary.
Cost: every agent interaction is a network round-trip with its own latency and failure modes.

🔌

Injector DI everywhere

Why: one codebase from laptop to cloud; backends are config, not forks.
Cost: indirection — “what runs?” is never answered at the call site.

📜

Event-sourced history

Why: replayable, streamable, and an open seam for extensions.
Cost: a pluggable store forces a lowest-common-denominator API across disk/S3/GCS.

🛡️

Resilient callbacks

Why: a buggy hook must never crash a user's agent run.
Cost: failures go quiet — an ERROR row no one watches.

🔒

Privacy by default

Why: consent-gated analytics, secret redaction in logs, OSS/SaaS split.
Cost: more branches and guards in every telemetry path.

Honest limitations — named, not hidden

A platform you can reason about is one whose weak spots you know. These are real, and some are visible right in the git history.

🔁
Callback dispatch is query-driven, per event

execute_callbacks runs a database SELECT for every event. At high event volume that's load the design hasn't yet optimised away — it's pull, not push.

⏳
Some reactions still poll

The shipped title-setter polls the agent server a fixed number of times with sleeps — pragmatic, but not event-driven, and it can give up before the title exists.

🧪
No in-repo evaluation framework

As Module 6 was blunt about: scoring lives in an external repo. The app server gives you data and a hook, not a harness.

📌
Tight coupling to the SDK version

The git log is full of bump agent-server to 1.x commits. Event kinds and behavior can shift under you; pin and test deliberately.

🏷️
Observability is vendor-shaped

Laminar for traces, PostHog for analytics — both fine choices, but neither is a vendor-neutral, injector-pluggable abstraction yet.

🔬
Evidence, not vibes

These aren't guesses — the recent history shows the team actively hardening exactly these seams (e.g. a fix to scope DB sessions tightly in poll_agent_servers to prevent idle-in-transaction). Mature systems carry known rough edges and a visible trail of sanding them down.

The road ahead

🔮
READ THIS FIRST — everything below is prediction

The cards on this screen are the author's informed speculation, not features of OpenHands today. None of this is in the codebase as of this course. Each is tagged NOT IMPLEMENTED so you never mistake a wish for a fact. Treat them as “natural next moves,” not a roadmap.

🔮

Vendor-neutral OTel exporter NOT IMPLEMENTED

Prediction: a first-class OpenTelemetry injector so traces flow to any backend without a custom callback — closing the layer-2 gap from Module 6.

🔮

Push-based callback dispatch NOT IMPLEMENTED

Prediction: an in-memory or queue-backed subscription bus to replace the per-event DB query — lower latency and load at scale.

🔮

First-class EvaluationService NOT IMPLEMENTED

Prediction: an injector that runs a scorer (tests / LLM-judge) over a trajectory in-product, making Module 6's sample a supported abstraction.

🔮

Pluggable AnalyticsService injector NOT IMPLEMENTED

Prediction: today analytics is constructed directly, not selected by an injector. Bringing it under the Module 3 pattern would let you swap PostHog for anything.

🔮

Callback retry/backoff policy NOT IMPLEMENTED

Prediction: configurable retries and dead-letter handling so a transient failure in an observability hook self-heals instead of leaving a silent ERROR row.

🔮

microVM / WASM sandbox backend NOT IMPLEMENTED

Prediction: a new SandboxServiceInjector using lighter isolation (Firecracker, gVisor, WASM) for faster, cheaper agent startup at fleet scale.
