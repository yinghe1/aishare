Notes as a Database

The obsidian-bases skill — teaching AI to turn your vault into queryable, live-updating views.

What if your notes could answer questions?

Imagine asking: "Show me all my books tagged #reading, sorted by status, with how many pages each has." Without Bases, you'd scroll through folders manually. With Bases, you get a live table that updates the moment you add a new book note.

Obsidian Bases turns your vault into something like a personal spreadsheet — but the rows are your notes, and the data is already in your frontmatter. The obsidian-bases skill teaches AI to build these views for you.

📝
Your notes have properties

Frontmatter like status, tags, date, author — all stored in every note you create with the markdown skill.

🔍
A .base file writes the query

Like a saved search — "give me all notes tagged #book where status != done, sorted by date."

📊
Obsidian renders it live

Open the .base file and see a table, card grid, or list — updated automatically as notes change.

Anatomy of a .base file

A base file is written in YAML — think of it as a form you fill out: "filter by these rules, display these columns, group by this property."

CODE 
filters:
  or:
    - file.hasTag("book")
    - file.hasTag("article")

formulas:
  status_icon: 'if(status == "done", "✅", "📚")'

views:
  - type: table
    name: "Reading List"
    order:
      - file.name
      - author
      - formula.status_icon
PLAIN ENGLISH

Start of filter rules — which notes to include

"Or" means include a note if ANY of these conditions are true

Include notes tagged #book

OR include notes tagged #article

Formulas section — computed columns you can invent

If the note's status is "done", show ✅, otherwise show 📚

Views section — how to display the results

Display as a table (other options: cards, list, map)

The table's name, shown as a tab in Obsidian

Which columns to show, left to right

Column 1: the note's filename

Column 2: the author property from frontmatter

Column 3: the computed status icon formula

Filters — the smart search that remembers itself

Filters are rules that decide which notes appear. Unlike a one-off search you type and forget, a filter in a base file is permanent — saved, reusable, and always up to date.

file.hasTag("book") Include notes with the #book tag
status == "done" Only notes where the status property equals "done"
status != "done" Exclude done notes — show everything in progress
priority > 3 Only high-priority items (where priority is a number)
file.inFolder("Books") Only notes inside the Books folder
file.mtime > today() - "7d" Notes modified in the last 7 days

Filters can be combined with and: (all must be true), or: (any can be true), and not: (exclude matches). Think of them like a smart shopping cart: AND adds conditions, OR broadens the search, NOT removes things you do not want.

Formulas — computed columns you invent

Formulas let you calculate values that do not exist yet in your frontmatter. Think of it like adding a new column to a spreadsheet that computes its value from other columns.

CODE 
formulas:
  days_until_due:
    'if(due, (date(due) - today()).days, "")'

  priority_label:
    'if(priority == 1, "🔴 High",
     if(priority == 2, "🟡 Med", "🟢 Low"))'

  days_old:
    '(now() - file.ctime).days'
PLAIN ENGLISH

Define custom computed columns here

Column named "days_until_due"

If the note has a due property, calculate days remaining; otherwise show blank

Column named "priority_label"

If priority is 1, show 🔴 High; if 2, show 🟡 Med; otherwise 🟢 Low

Column named "days_old"

Subtract the file creation time from now, convert to days

⚠️
Common gotcha: date subtraction returns a Duration, not a number

When you subtract two dates in a formula, you get a Duration object — not a plain number. Always add .days to get the number: (date(due) - today()).days not just (date(due) - today()). This is the #1 formula error the skill prevents.
