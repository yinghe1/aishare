Your own machine. Direct access to your files and environment. Default for most developers getting started.

The Cron Scheduler — Standing Orders

A cron job is a task that fires on a schedule — every morning, every hour, once a week. Hermes has a built-in cron scheduler that accepts natural language prompts instead of cryptic time codes.

1
You give a standing order in chat

“Every morning at 8am, summarize my unread emails and send the digest to my Telegram.” No time codes, no config files — just plain English.

2
Hermes registers the cron job

The scheduler stores your prompt, the trigger time, and the target platform. It persists across restarts — the order stands until you cancel it.

3
Scheduler fires at 8am

The scheduler wakes up and starts a fresh, isolated agent session — completely separate from any conversation you might be having at the time.

4
Agent runs the task and sends results

The session executes your prompt, reads emails, writes the summary, and delivers it via the Gateway to your Telegram account.

🔒
Cron prompts are scanned for injection attacks before running.

If someone tricks you into creating a malicious cron job — by embedding a hidden instruction in a document you process — the scanner catches it before the scheduled session fires. This is prompt injection defense built into the scheduling layer.

🔵
Each cron job gets its own isolated session.

It runs independently — not inside your current conversation. Think of it as Hermes clocking in for a separate shift to handle the standing order, then clocking out when done.

What Children Are Not Allowed to Do

Subagents get a restricted toolset. Certain tools are always blocked — hardcoded, not configurable. Here is the actual code from delegate_tool.py:

CODE 

DELEGATE_BLOCKED_TOOLS = frozenset([
    "delegate_task",  # no recursive delegation
    "clarify",        # no user interaction
    "memory",         # no writes to shared MEMORY.md
    "send_message",   # no cross-platform side effects
    "execute_code",   # reason step-by-step, not scripts
])

_DEFAULT_MAX_CONCURRENT = 3
MAX_DEPTH = 2  # parent(0) -> child(1) -> REJECTED(2)
PLAIN ENGLISH

This is an immutable set — a permanent list of blocked tools that nobody can override, not even you.

delegate_task blocked: Children cannot spawn their own children. No recursive armies — the hierarchy stops at parent → child.

clarify blocked: Children cannot ask you questions. They work silently and independently. If stuck, they report failure to the parent.

memory blocked: Children cannot write to your shared MEMORY.md or USER.md. Only the parent agent can update shared memory — prevents race conditions.

send_message blocked: Children cannot send messages to Telegram or Discord. They cannot pretend to be you.

execute_code blocked: Children reason step-by-step rather than writing and running scripts. Keeps behavior predictable and auditable.

MAX_DEPTH = 2: The nesting limit. Parent is depth 0, child is depth 1, any grandchild attempt is immediately rejected at depth 2.

Delegation Safety

Parallel agents are powerful — and dangerous if they can step on each other. Hermes enforces strict isolation rules so children stay focused and can’t corrupt shared state.

🚫
No recursive delegation

Children cannot spawn grandchildren. The delegation tree is a flat two-level structure: parent and children. No infinite armies, no runaway compute.

🔇
No user interaction

Children work silently. They cannot ask you questions or receive guidance mid-task. If a child is stuck, it reports failure — the parent then tells you what went wrong.

🔒
No shared memory writes

Children cannot write to MEMORY.md or USER.md. Multiple agents writing shared data simultaneously causes race conditions — so it is simply blocked.

📵
No message sending

Children cannot send messages on your behalf through the Gateway. Only the parent agent can communicate externally — prevents impersonation and unexpected side effects.

📐
MAX_DEPTH = 2: parent → child → grandchild rejected.

The depth counter is passed to each child. When a child tries to delegate, the depth check fires and the request is refused immediately — no matter what instructions it was given.

Glossary

gateway

The always-on listener that connects Hermes to messaging platforms like Telegram, Discord, and Slack.

webhook

An instant push notification from one service to another. Telegram uses webhooks to tell Hermes about new messages immediately.

cron

Standing orders that fire on a schedule. Hermes accepts natural language — no time code syntax required.

daemon

A background process running silently at all times — the Gateway and scheduler are both daemons.

subagent

A child agent with an isolated context and restricted toolset. Reports back when done — never mid-task.

serverless

Hibernates when idle, wakes when needed. You pay only for compute time actually used. Daytona and Modal are both serverless.

terminal backend

Where Hermes runs commands. Six options: local, docker, ssh, daytona, singularity, modal.
