40+ Powers — The Toolkit

Every tool Hermes wields, how toolsets control access, and the safety gate that protects you from dangerous commands

The Tool Zoo

When you ask Hermes to build a REST API, it doesn’t guess what commands to run. It has 40+ specific tools — each purpose-built for exactly one kind of task. Think of it like a Swiss Army knife: every blade is specialized, retractable when not needed, and there’s a safety lock for the dangerous ones.

💻

Terminal Tools

Run shell commands, manage background processes, interact with git, and execute scripts inside any configured runtime environment.

📄

File Tools

Read, write, patch, and search through your entire codebase. These are the most-used tools in any coding session.

🌐

Web Tools

Search the internet, browse any URL, and extract structured content from web pages. Hermes can read documentation, Stack Overflow answers, and live APIs without leaving your terminal.

🧠

Memory Tools

Read and write to MEMORY.md and USER.md. This is how Hermes builds up knowledge about you and your projects over time.

📋

Skills Tools

List, view, and invoke skills. Skills are Hermes’ learned knowledge — procedures it wrote for itself after solving hard problems.

🤖

Delegation

Spawn and manage subagents to work on sub-tasks in parallel. One Hermes can become many, each focused on a different piece of your request.

⏰

Cron Tools

Create and manage scheduled tasks. Tell Hermes to run a report every morning, check a price every hour, or back up files every night.

🎤

Voice & Vision

Transcribe audio files and analyze images. Drop a voice memo or screenshot into a session and Hermes reads it the same way it reads a text file.

🔌

MCP Tools

Connect to external services like Google Drive, Gmail, and GitHub via the Model Context Protocol. Each connected service appears to Hermes as a set of native tools.

📦
Where tools live in the code

All tools are registered in tools/registry.py. The agent selects which ones to load based on the active toolset configuration. You never need to touch this file — but knowing it exists tells you exactly where to look when you want to understand what Hermes can do.

Toolsets — The Permission System

Tools are grouped into toolsets — named permission tiers. Instead of toggling 40 individual tools, you enable or disable entire categories. It’s the difference between handing someone a whole toolbox vs. one specific wrench.

terminal Run shell commands, manage processes, execute scripts in the runtime environment
file Read, write, patch, and search files in the local filesystem
web Search Google, browse URLs, extract structured content from web pages
memory Read and write to MEMORY.md and USER.md — Hermes’ long-term notes
skills List, view, and invoke saved skill procedures from the skills library
delegation Spawn subagents to handle sub-tasks in parallel — the multiplier tool
moa Mixture-of-agents mode — route sub-tasks to multiple AI models simultaneously for higher-quality results
🔒
Subagents get a restricted toolset

When Hermes delegates work to a subagent, that subagent receives a narrower toolset: terminal, file, and web only. No delegation (subagents cannot spawn further subagents by default), no memory writes, no moa. This is a deliberate safety boundary — controlled recursion, not unlimited sprawl.

💡
Why this matters to you

Knowing the toolset system means you can tell Hermes exactly what it’s allowed to touch — and what it isn’t. Running Hermes in a read-only audit mode? Disable terminal, file writes, and delegation. Running an automated pipeline? Disable memory writes so it doesn’t pollute your notes with automation artifacts.

The Dangerous Command Gate

Before Hermes runs any shell command, it checks it against a list of dangerous patterns. If the command matches, execution halts and you get to decide what happens next. This is your safety net.

CODE — tools/approval.py 

DANGEROUS_PATTERNS = [
  (r'\brm\s+(-[^\s]*\s+)*/',
    "delete in root path"),
  (r'\brm\s+-[^\s]*r',
    "recursive delete"),
  (r'\brm\s+--recursive\b',
    "recursive delete (long flag)"),
  (r'\bchmod\s+(-[^\s]*\s+)*'
   r'(777|666|o\+[rwx]*w|a\+[rwx]*w)\b',
    "world/other-writable permissions"),
]
PLAIN ENGLISH

DANGEROUS_PATTERNS is a list of pairs: each pair is a regex pattern and a human-readable description of why it’s dangerous.

The first three patterns catch variations of rm -rf — the delete-everything command. The \b anchors mean the pattern only matches the word rm, not a longer word like prompt.

The chmod 777 pattern catches attempts to make files world-writable — a common security mistake that opens files to any user or process on the system.

There are 40+ patterns in total, covering: SQL DROP/DELETE, disk formatting, fork bombs, piping remote scripts to shell, git force pushes, and self-termination commands. Every real danger has a named pattern.

Three Approval Levels

✅
Approve Once

Allow this specific command right now, for this one invocation only. Next time the same command appears, Hermes will ask again.

🔓
Approve for Session

Allow this pattern for the rest of this conversation. When you close the session and start a new one, the approval resets. Useful for repetitive workflows in a single work session.

📝
Add to Allowlist

Save this pattern permanently to config.yaml. Hermes will never ask about it again, across all future sessions. Use this for commands you run all the time in automated workflows.

🤖
Smart approval — the auto-approver

Hermes uses a fast secondary AI model to pre-screen flagged commands. If the model determines the command is clearly low-risk in context, it auto-approves silently. You are only interrupted when the command is genuinely ambiguous or dangerous — not for every false alarm.

MCP — Connect to Anything

The Model Context Protocol turns Hermes into a universal connector. Google Drive, Gmail, GitHub, Notion, Slack — if there’s an MCP connector for it, Hermes can read and write it.

1
You enable a connector

Add the connector to your config — for example, the Google Drive MCP server. Hermes discovers it automatically on next startup.

2
Hermes guides you through OAuth login — once

The first time you use a connected service, Hermes walks you through authenticating on that service’s own website. You log in directly — Hermes never sees your password.

3
Credentials are stored securely

The access token is saved in Hermes’ local credential store. Future sessions reuse it automatically — no re-login needed until the token expires.

4
Google Drive appears as native tools

Hermes now has tools like gdrive_list_files, gdrive_read_doc, gdrive_create_doc. They show up exactly like built-in tools — no special syntax needed.

5
Hermes reads, searches, and writes your Drive

Ask "find my Q3 budget spreadsheet and summarize it" — Hermes searches Drive, reads the file, and replies. All in one message, same as any other task.

🔍
The AI cannot tell the difference

MCP tools look exactly like built-in tools to the AI model. The model does not know or care whether it’s reading a local file or a Google Doc — it just receives the content as text. This means every skill Hermes has for analyzing code or summarizing documents works equally well on MCP-sourced data.
