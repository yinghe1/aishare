Inside Codex CLI
01

The Prompt's Journey

You type codex. Something starts up. But which something? Same binary, six different destinations — let's walk through the terminal.

First, what even is Codex?

Codex is an AI coding agent that runs on your own laptop. It reads your files, edits them, and runs shell commands — all under permissions you control. The whole thing is one binary called codex.

✈️
The metaphor for this module: an airport terminal

The codex binary is one building. Each subcommand (exec, mcp-server, sandbox, …) is a different gate that leads to a completely different journey. Same building. Totally different flights.

Same word, two very different worlds

Type codex on its own and you get a chat box. Type codex exec "fix the bug" and there's no chat at all — just answers piped straight to your terminal. How does the same program know which one you wanted?

💬
You type: codex

A full-screen chat opens up. You talk to it. It talks back. This is the TUI mode.

⚡
You type: codex exec "fix the bug"

No UI. Codex thinks, acts, prints the result, and exits. This is headless mode — perfect for scripts.

💡
Key Insight: Codex is a polymorphic binary

One executable that becomes whichever sub-program you ask for. The very first thing it does — before anything model-related — is decide which version of itself to be. Engineers call this polymorphic behavior.

The six gates of the terminal

Most vibe coders only ever discover gate #6 — the chat. The other five are where the real power lives. Here's each one and where it takes you.

⚡

codex exec

Run Codex once, headless. Give it a prompt, get a result, exit. The gate for shell scripts and CI pipelines.

🔌

codex mcp-server

Become a server that other AI tools (Claude Code, Cursor) can call as a tool. MCP = AI talking to AI.

🔐

codex sandbox

Test any shell command inside the sandbox jail — no AI involved. Pure permissions playground.

📝

codex apply

Take the diff Codex last suggested and run git apply on it. The "okay, do it for real" gate.

📚

codex resume

Reopen a past conversation thread. Codex remembers where you left off so you can keep going.

💬

codex (default)

No subcommand — you land in the full chat interface. This is the gate almost everyone uses.

The receptionist that routes you

So how does the binary actually decide? It uses a little Rust library called clap to read what you typed and match it against a list of enum variants. Each variant is a gate.

CODE · codex-rs/cli/src/main.rs 
#[derive(Debug, clap::Subcommand)]
enum Subcommand {
    #[clap(visible_alias = "e")]
    Exec(ExecCli),               // non-interactive one-shot
    Mcp(McpCli),                 // MCP server management
    McpServer(McpServerCommand), // become an MCP server
    Sandbox(HostSandboxArgs),    // sandboxed exec wrapper
    Apply(ApplyCommand),         // git apply last diff
    Resume(ResumeCommand),       // reopen past thread
    // ... plus AppServer, Cloud, Debug, Plugin, Doctor, etc.
}
PLAIN ENGLISH

"Hey clap, this enum is a list of subcommands — please match the user's input against it."

Define the list of possible gates the user might walk through.

Let the user type just e as a shortcut for exec — the receptionist accepts both forms.

Gate 1: headless one-shot. You ask, it answers, it leaves.

Gate 2: manage which external MCP servers Codex itself talks to.

Gate 3: become an MCP server so other agents can call Codex.

Gate 4: run a plain shell command inside the sandbox — no AI involved.

Gate 5: take the most recent suggested diff and apply it to your git repo.

Gate 6: pick up an old thread where you left off.

Trailing comment: there are more gates we'll explore later (cloud, debug, doctor…).

Close the enum — that's the full menu of gates.

What you see in the docs is just the public face of this same enum — the friendly side of the receptionist's clipboard:

"mcp-server" = flip the chairs around. Other AI tools now call you.

"sandbox + a command" = test what the jail will and won't allow.

What actually happens when you hit Enter

Trace one keystroke from your fingertips all the way to whichever gate you end up at. Five steps, in order:

1
You type the command and press Enter

Your shell hands the binary its argv — the list of words you typed, starting with the program's own name (arg0).

2
Codex peeks at its own name first

Before parsing anything, it checks arg0. If it was invoked as codex-linux-sandbox, it reroutes itself into the sandbox entrypoint — the famous "arg0 dispatch" trick. Same binary, different personality.

3
clap parses the rest of your words

The receptionist library reads "exec" or "mcp-server" or sees nothing at all, and matches it against the Subcommand enum from the last screen.

4
The matching gate is selected

Now Codex knows which version of itself to become — headless runner, MCP server, sandbox tester, diff applier, thread resumer, or the chat TUI.

5
That gate's run() boots its own world

Each subcommand has its own startup routine. TUI builds a chat interface. exec opens a pipe. mcp-server starts listening on a socket. The model is loaded only after the gate decides it actually needs one.

🔍
Why this matters for you

Most users never get past step 3 because they never type a subcommand. Knowing the gates means you can drop Codex into scripts (exec), wire it into other agents (mcp-server), or test sandbox rules without burning model tokens (sandbox). Same superpower, different doorway.

Pick the right gate

Three scenarios. For each one, which subcommand would you reach for? No memorizing — just think about which gate fits the journey.

You want to run Codex inside a shell script that fixes a build error and then exits. Which gate?

You want Claude Code or Cursor to be able to call Codex as if Codex were one of their built-in tools. Which gate?

You want to test whether rm -rf node_modules would be blocked by the sandbox rules — without launching the model at all. Which gate?

02

Meet the Actors

We walked through the chat-mode gate. Now we’re inside the building — and inside, five characters are doing all the work. Once you know their names, the wall of folders turns into a cast list.

Codex is a jazz ensemble, not a soloist

Think of a small jazz group. The bassist holds the rhythm. The drummer cues changes. The horn plays the melody. Nobody plays every instrument — the music comes from how they listen to each other.

Codex is built the same way. It’s not one giant program; it’s a federation of small crates that pass notes to each other. None of them is in charge alone.

🎧
The opposite of a monolith

A monolith is one big tangled program where editing the chat UI could accidentally break the AI model logic. Codex is the opposite: each crate has one job, one job only, and refuses to do anyone else’s.

The rehearsal room: codex-rs/

If you open the Codex source code, the Rust side of the project lives in one folder called codex-rs/. Inside, you’ll find a handful of crates — each one a member of the band.

codex-rs/ The Rust workspace — where the band rehearses
cli/ Reads what you typed at the terminal and picks which Codex to launch
tui/ The fullscreen chat box you actually look at — draws every pixel
core/ The engine that talks to the AI model and runs tool calls
exec/ A headless version for automation — same engine, no chat box
apply-patch/ The crate that actually rewrites your files when the model proposes edits
mcp-server/ Lets Codex be controlled by other AI tools through the Model Context Protocol
linux-sandbox/ The safety cage that wraps shell commands on Linux

Each folder is one crate. They live in the same workspace, but they only talk to each other through narrow, well-defined doors. That’s the whole secret of how Codex stays understandable.

The five actors

Forget folders for a second. Here are the five characters that actually do the work, with the role each one plays in every Codex session.

🏫
The Receptionist — cli/

Reads the command you typed and decides which version of Codex to launch — chat mode, headless mode, or something else. The receptionist greets you, then hands you off to the right room.

🎪
The Stage Manager — tui/

Draws the chat box, captures every keystroke, displays streamed messages from the model. Built with Ratatui. Crucially: the Stage Manager does not call the AI model. Ever.

🎤
The Conductor — core/

Runs the turn loop (Module 3’s star), talks to the AI model, decides which tool to call next, and dispatches the work. The Conductor does not draw a single pixel on your screen.

📝
The Stagehand — exec/

Same engine as the Conductor, but pipes results straight to plain stdout instead of a fancy chat UI. Used when Codex is wired into scripts, CI pipelines, or other automation.

🛡️
The Bouncer — linux-sandbox/, windows-sandbox-rs/, seatbelt

Sits between the model’s shell commands and your real operating system. If the model says “delete this folder,” the Bouncer asks: is that allowed? We’ll meet the Bouncer in depth in Module 5.

How they talk: events on a channel

So if the Stage Manager can’t call the model, and the Conductor can’t draw the screen — how does anything get done?

They pass notes. Specifically, events on a channel — like musicians watching the conductor’s baton instead of whispering to each other. Here’s the actual list of things the TUI is allowed to tell the rest of Codex:

CODE 
#[allow(clippy::large_enum_variant)]
#[derive(Debug)]
pub(crate) enum AppEvent {
    /// Open the agent picker for switching active threads.
    OpenAgentPicker,
    /// Switch the active thread to the selected agent.
    SelectAgentThread(ThreadId),

    /// Fork the current thread into a transient side conversation.
    StartSide {
        parent_thread_id: ThreadId,
        user_message: Option<UserMessage>,
    },
    /// Submit an op to the specified thread, regardless of current focus.
    // ...
}
PLAIN ENGLISH

This is the complete list of things the chat UI can ask the engine to do. Nothing else. If it’s not on this list, it doesn’t happen.

An enum means “one of these specific options.” AppEvent is the menu of possible notes the Stage Manager can pass.

Variant 1: “Open the menu for switching between different agent conversations.”

Variant 2: “Switch to this specific conversation thread.” (The ID inside the parentheses says which one.)

Variant 3: “Start a side conversation branched off this one” — useful for trying an idea without polluting the main thread.

And the list keeps going. Every button, every shortcut, every message ultimately becomes one of these events fired down the channel.

This is what software engineers mean by message passing. The Stage Manager doesn’t reach inside the Conductor and pull levers. It writes a note, drops it on the conveyor belt, and trusts the Conductor to read it.

A wall, not a guideline

Here’s the punchline. The Conductor’s code starts with a single line that makes the whole arrangement non-negotiable:

CODE 
//! Root of the `codex-core` library.

// Prevent accidental direct writes to stdout/stderr in library code. All
// user-visible output must go through the appropriate abstraction (e.g.,
// the TUI or the tracing stack).
#![deny(clippy::print_stdout, clippy::print_stderr)]
PLAIN ENGLISH

The two slashes-bang at the top is just a header comment naming the file: “this is the root of the core library.”

The next comment is the team telling future-them why: nobody inside the engine is allowed to print directly to the terminal. All output must flow through the proper channel.

The last line is the enforcement. #![deny(...)] tells the compiler: refuse to build this code if anyone calls println! or eprintln! in here. Not a warning — a hard error.

clippy is the strict editor doing the yelling; print_stdout and print_stderr are the two lints being escalated from warning to wall.

💡
Aha! — the wall is the architecture

The Conductor cannot print to your terminal — even if a developer forgets and tries. The compiler refuses to build it. Every word you see in the chat had to be sent as an event the Stage Manager chose to display. Engineers call this separation of concerns: give each piece one job, and one job only. Most projects treat this as a polite suggestion. Codex bakes it into the build.

Why this map matters

When something goes wrong with Codex, this cast list is your first debugging tool. The symptom tells you which actor to suspect.

🎪

UI frozen, files still changing

The Stage Manager is stuck. The Conductor is fine — it’s just that no one is updating the screen.

🎤

File written, no chat message

The Conductor did the work, but the event announcing it never reached the Stage Manager. The channel got dropped.

🛡️

“Permission denied” on a shell command

The Bouncer blocked it. Nothing to do with the model — it’s the safety actor doing its job.

So who keeps the music going? What tells the Conductor when to count off the next bar, when to call the saxophone in, when to fade out? That’s the turn loop — the heartbeat that drives every Codex conversation. Meet it in Module 3.

Module 3

The Turn Loop

We met the actors. Now meet the music — the loop that drives every turn.

You ask Codex to “fix the failing test.” Sometimes the answer arrives in 5 seconds. Sometimes it grinds for 5 minutes — editing files, running commands, thinking again. Why the difference?

Because under the surface, that one prompt is not one round-trip. It is a loop.

📻
The metaphor: a two-way radio

Picture a ham-radio conversation. You press the button and speak (the prompt). The other person starts replying while still thinking, broadcasting word by word (streaming). You can break in any time. Sometimes they pause to look something up (a tool call) before continuing. The call only ends when one of you says “over and out.”

The loop, in code

Here is the heart of core/src/session/turn.rs, distilled into pseudocode. The shocking thing for most people: this is a loop. Most users think it is one call.

CODE 
loop {
  // 1. Pick up any new user input that arrived while we were thinking
  pending_input = drain_input_queue();

  // 2. Build the prompt: history + tools + personality
  prompt = build_prompt_from_history();

  // 3. Open a stream to the model and start receiving events
  stream = client.stream(prompt, model_info).await;

  // 4. Pull events one by one until the stream ends
  while let Some(event) = stream.next().await {
    handle(event);  // could be a text token, a function call, etc.
  }

  // 5. Did the model finish, or does it want to run a tool and continue?
  if !needs_follow_up { break; }
}
PLAIN ENGLISH

1. Before we talk to the model, scoop up anything new you typed while it was thinking. Codex never makes you wait to type.

2. Bundle the whole conversation so far (your messages, the model’s replies, tool results) plus the list of tools it’s allowed to use. That bundle is “the prompt.”

3. Open a live radio channel to the model. We are not asking for one big answer — we are asking it to start broadcasting.

4. Loop over every little event as it arrives: a token, a function call, a status update. Each one gets handled the moment it shows up.

5. When the stream ends, check: was that the final word, or does the model want a tool run so it can keep going? If it’s done, exit. Otherwise, loop again with the new info.

💡
Aha! moment: streaming is real

You see text appearing letter by letter not because of an animation. The model is literally sending tokens one at a time over a network connection, and the TUI is rendering each one the moment it arrives. The “typing” effect is real — which is also why you can interrupt mid-stream.

When (and how) the loop stops

Two questions that explain 90% of Codex’s behavior: who decides when to stop? and what if the stream breaks mid-broadcast?

CODE — client_common.rs 
pub struct ResponseStream {
    pub(crate) rx_event: mpsc::Receiver<Result<ResponseEvent>>,
    pub(crate) consumer_dropped: CancellationToken,
}

impl Stream for ResponseStream {
    type Item = Result<ResponseEvent>;
    fn poll_next(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Self::Item>> {
        self.rx_event.poll_recv(cx)
    }
}
PLAIN ENGLISH

A ResponseStream is a little pipe with two ends. One end (mpsc) is for receiving events.

The other end is a CancellationToken — basically a kill switch. When you hit Ctrl-C, this flips, and any task watching it stops gracefully.

The poll_next function is how the loop asks “got anything new for me yet?” This is the radio click between sentences.

No tokens? Quiet, but the line stays open. Tokens? Hand them up to the loop and keep listening.

And if the radio cuts out?

CODE — turn.rs 
let event = match event {
    Some(Ok(event)) => event,
    Some(Err(err)) => break Err(err),
    None => {
        break Err(CodexErr::Stream(
            "stream closed before response.completed".into(),
            None,
        ));
    }
};
PLAIN ENGLISH

Three things can come down the pipe…

A real event: great, use it.

An error: stop the loop and bubble the error up.

Nothing (None): the connection went dead before the model said “done.” That is an error too — Codex throws stream closed before response.completed instead of pretending everything is fine.

💡
Aha! moment: the loop ends when the model says so

There is no hard-coded turn limit. The model decides when its answer is final. That is why a complex task can run for many round-trips — and why setting a max-turns config matters when you want to bound runaway behavior.

Module 4

Tools & Approvals

Last module we watched the model “call a tool.” What does that even mean?

When you ask Codex to “rename this variable,” the model does not reach into your filesystem. It writes a small request: please use the apply_patch tool with this diff. Then someone — you, or a policy — decides whether the key turns.

🔑
The metaphor: a locksmith with a keyring

The model is a locksmith standing outside your house. It cannot just walk in. It has to pick a key (a tool) and ask permission to turn it. Some keys open anywhere (reading a file). Some need the homeowner’s nod (editing a file). Some doors are bolted shut and no key works (writing outside the workspace under read-only mode).

Tools are named contracts. Every action the model takes is one of a small, finite list of named tools, each with structured arguments. Approval is layered: first the policy decides, then a safety check decides, then maybe you decide. Three filters before a key turns.

What is on the keyring?

Each card below is one key. When the model emits a function call, it is naming one of these — never anything else.

💻

shell

Run a command inside your workspace — like pnpm test or git status. The biggest, most powerful key on the ring.

✏️

apply_patch

Edit one or more files atomically using a unified diff.

📄

read_file

Pull the contents of a file into the conversation. In practice often expressed as a shell call to cat.

🌐

web_search

Look something up on the open internet — for fresh docs, error messages, library versions Codex was not trained on.

🔌

MCP tools

Custom keys you add to the ring via MCP servers — database queries, ticketing systems, your own scripts, whatever you wire up.

Notice what is not here: there is no “do anything you want” key. The model can only ask for things on this list. If you wanted Codex to send a Slack message, you would have to add a Slack MCP server — a new key.

The apply_patch trick

This is the most elegant hack in the whole codebase. Once you see it, you will spot it everywhere — and you will know when an LLM is faking it.

How do you let an AI edit a file safely? Codex’s answer: don’t invent a new format — use bash. The model emits a regular shell command that embeds a diff inside a heredoc.

CODE — what the model emits 
apply_patch <<'EOF'
*** Begin Patch
*** Update File: src/parser.ts
@@
-  const valeu = config.value;
+  const value = config.value;
*** End Patch
EOF
PLAIN ENGLISH

1. Call the apply_patch command. Everything between the two EOF markers is one giant string argument.

2. “Here comes a patch.”

3. “The file I want to change is src/parser.ts.”

4. A diff marker — like a chapter heading inside the patch.

5. The minus line: “remove this typo, valeu.”

6. The plus line: “replace it with value.”

7. “End of patch.”

8. Close the heredoc — the shell stops swallowing text and runs the command.

Codex parses that heredoc with tree-sitter (a real bash parser), validates the diff format, and applies it atomically. Three reasons this is brilliant:

💡
Why the heredoc is genius

(a) The model gets to reason about its edit as plain text — a diff is just words. (b) It looks like a normal shell command, so the same approval pipeline that handles every other shell call also handles patches — no separate code path. (c) Shell escaping does the hard quoting work for free. One trick, three problems solved.

Three filters before the key turns

Even with a valid patch, the locksmith still has to ask. Inside Codex, every apply_patch goes through a tiny pipeline that decides: auto-approve, ask the user, or refuse outright.

The result of that decision is shaped like this — a Rust enum with two variants:

CODE — apply_patch.rs 
pub(crate) enum InternalApplyPatchInvocation {
    /// The `apply_patch` call was handled programmatically, without any sort
    /// of sandbox, because the user explicitly approved it. This is the
    /// result to use with the `shell` function call that contained `apply_patch`.
    Output(Result<String, FunctionCallError>),

    /// The `apply_patch` call was approved, either automatically or explicitly.
    /// The runtime realizes the patch through the selected environment filesystem.
    DelegateToRuntime(ApplyPatchRuntimeInvocation),
}

#[derive(Debug)]
pub(crate) struct ApplyPatchRuntimeInvocation {
    pub(crate) action: ApplyPatchAction,
    pub(crate) auto_approved: bool,
    pub(crate) exec_approval_requirement: ExecApprovalRequirement,
}
PLAIN ENGLISH

Variant 1 — Output: We already ran the patch ourselves (because you explicitly approved it earlier). Here is the result — success message or error — ready to hand back to the model.

Variant 2 — DelegateToRuntime: The patch is approved. Hand it to the runtime, which will apply it through the sandbox that’s currently active. We do not run it ourselves.

The struct underneath just carries the details: the parsed patch, whether it was auto-approved, and what approval level the sandbox should treat it as.

And here is the actual decision — a three-way switch with three possible verdicts. This single match block is the whole approval engine.

CODE — assess_patch_safety 
match assess_patch_safety(
    &action,
    turn_context.approval_policy.value(),
    &turn_context.permission_profile(),
    file_system_sandbox_policy,
    &action.cwd,
    turn_context.windows_sandbox_level,
) {
    SafetyCheck::AutoApprove { .. } => /* run it */,
    SafetyCheck::AskUser => /* show approval card to user */,
    SafetyCheck::Reject { reason } => /* tell the model: nope, denied */,
}
PLAIN ENGLISH

The call to assess_patch_safety: hand it the patch, your approval policy, your permission profile, the sandbox rules, and the working directory. It thinks — then returns a verdict.

Verdict A — AutoApprove: “Just do it.” The policy says this is safe enough to run without bothering you.

Verdict B — AskUser: “Show them the approval card.” The TUI pops up the diff and waits for you to click Allow or Deny.

Verdict C — Reject: “Nope, denied.” The reason gets sent back to the model as the tool output — so the model can adjust and try a different approach.

Three policies, three trust levels

You get to pick how loud the locksmith knocks. This is set with --ask-for-approval on the command line, or in your config file. Three settings — biggest config knob you have.

never Auto-approve everything. The model can run anything the sandbox policy allows. Use when you trust the agent and have a tight sandbox.
on-request Every tool call pops up an approval card. You click Allow or Deny each time. Maximum paranoia, maximum interruptions.
unless-trusted Codex auto-approves if the command’s “shape” matches a known-safe pattern (like ls or cat). Otherwise it asks. The default sweet spot.

Aha! moment: errors flow back as feedback

When a tool fails — bad patch, missing file, permission denied — Codex does not crash or panic. Engineers call this failing forward: every error is just more information for the next loop iteration.

Next up: but what if a tool itself misbehaves?

Approval is one filter — you (or a policy) deciding before the key turns. But what if the model tricks the policy, or a tool has a bug, or an MCP server you trusted goes rogue? You need a wall around the locksmith, not just a checkpoint in front of him. That is the sandbox. Module 5.

Module 5

The Sandbox

Module 4 ended with a question: approval is one filter — what about the sandbox? Here is the layer that stands between a confused model and your home directory.

Codex runs shell commands. Real ones. On your real machine. Have you ever wondered why there aren’t a million horror stories of agents nuking people’s home folders? Because of this layer.

🧪
The metaphor: a glass-walled cleanroom

Picture a cleanroom inside a chip factory. The technician inside can see the rest of the building through the glass — read manuals, look at samples — but contaminants can’t leak out. Sealed doors, filtered air, locked exits. The agent works inside this room. macOS, Linux, and Windows each build the room differently, but they all do the same job: contain the work.

Crucially, the cleanroom is enforced by the building, not by the technician’s good behavior. The agent doesn’t politely refrain from misbehaving — it literally cannot. The kernel refuses.

Three sizes of cleanroom

Codex offers exactly three settings — one safest, one default, one break-glass. This is the most important config choice you’ll make.

👀

read-only

Can read any file. Can’t write anything. No network. The safest setting. Use when you want the agent to explore and propose, not act.

✏️

workspace-write

Can read anything. Can write only inside the project folder. No network unless explicitly allowed. The default. Best for most coding work.

🔥

danger-full-access

No filesystem or network restrictions. Reserve for inside Docker containers or virtual machines. The README literally calls it “Danger!”

The three levels show up everywhere in the codebase — in the CLI, in config.toml, in the internal policy types. One unified abstraction; three OS-specific implementations underneath.

Three operating systems, three cleanrooms

Same policy. Same levels. Three completely different mechanisms to actually enforce the walls — one per OS.

🍎

macOS — Seatbelt

Seatbelt (sandbox-exec) reads an Apple profile language describing what’s permitted. The kernel enforces it. Inside, CODEX_SANDBOX=seatbelt is set so child code knows it’s caged.

🐧

Linux — Landlock + bubblewrap

Landlock restricts which paths a process can touch. bubblewrap handles the surrounding cage. A helper binary, codex-linux-sandbox, wraps the actual command.

🪟

Windows — Restricted Token

Two modes — Elevated and RestrictedToken — controlled by a restricted token, Windows’s way of saying “this process has fewer permissions than you do.”

The whole point of three implementations is that the rest of Codex doesn’t care. It just says “spawn this sandboxed” — the right cleanroom gets built underneath.

How Codex picks the cleanroom

Here is the Windows side, in real Codex source. Notice how short it is — the whole “which cage do we use?” decision fits on one screen.

CODE — windows_sandbox.rs 
impl WindowsSandboxLevelExt for WindowsSandboxLevel {
    fn from_config(config: &Config) -> WindowsSandboxLevel {
        match config.permissions.windows_sandbox_mode {
            Some(WindowsSandboxModeToml::Elevated) => WindowsSandboxLevel::Elevated,
            Some(WindowsSandboxModeToml::Unelevated) => WindowsSandboxLevel::RestrictedToken,
            None => Self::from_features(&config.features),
        }
    }
}
PLAIN ENGLISH

Given the user’s config, decide which Windows cage to build.

Config says Elevated? Use the elevated cage — more privileges, used for setup-style tasks.

Config says Unelevated? Use the restricted-token cage — the tighter, more locked-down one.

No config at all? Fall back to picking based on feature flags. Codex always has some answer.

💡
Aha! moment: defense in depth

The sandbox does not trust the agent — it trusts the operating system. Even if the model is malicious or buggy, even if the tool runtime has a bug, the kernel still refuses the syscall. The safety isn’t a check the agent runs — it’s a wall it can’t see through. Engineers call this defense in depth.

Watch the wall hold

A bad command comes in. The model wants to delete the user’s home directory. Approval lets it through. Watch what happens at the next layer.

🧠
Model

⚙️

🧪

Denied. Write attempt outside allowed paths.

🧠
Model

Acknowledged. Trying a safer approach.

?

The model didn’t “learn its lesson” through training. The error came back like any other tool result — and the loop kept going. The error is the feedback.

The dials you actually touch

Three flags and two env vars explain almost every “why can’t Codex do this?” moment.

User-facing CLI flags

CODE — README.md 
codex --sandbox read-only           # default — explore freely, can’t write
codex --sandbox workspace-write     # write inside the project, no network
codex --sandbox danger-full-access  # break glass — only inside containers
PLAIN ENGLISH

Three commands, three cleanrooms. Same Codex, different walls.

read-only: “Look but don’t touch.” Great for code review and exploration.

workspace-write: “Edit my project, leave my home alone.” The everyday default.

danger-full-access: “I built the room around you — you don’t need your own.” For Docker, CI, throwaway VMs.

Env vars and config you’ll spot in the wild

CODEX_SANDBOX=seatbelt Set inside Seatbelt’d processes so code can detect it’s caged. Tests use it to skip operations the sandbox would block (you can’t nest Seatbelt).
CODEX_SANDBOX_NETWORK_DISABLED=1 Set when network is blocked. Tests check this and skip anything that needs the internet.
--sandbox <level> CLI flag that overrides the configured default for this single run.
[permissions] in config.toml The persistent way to set the default sandbox level — per machine, or per project. This is what permission profiles are built from.

Module 6 · Finale

Memory — how a conversation survives

Module 5 ended with a question: how does Codex remember? Answer: it keeps a logbook.

Close your terminal. Walk away. Come back tomorrow. Type codex resume — and the conversation picks up exactly where you left it, with the agent remembering every file it read, every patch it wrote, every decision the two of you made. That is not magic.

📖
The metaphor: a ship’s logbook

Picture an old sailing ship. Every shift, the officer on watch writes one line in a bound book: 14:02 — spotted reef to port. 14:05 — turned hard starboard. 14:18 — clear water. Pages are never torn out. If the ship sinks, the next crew reads the log and knows what happened. If the book gets too thick to carry, someone writes a summary page so the captain doesn’t have to read the whole thing on every shift. That is Codex’s memory, exactly.

Three layers of memory

Codex has memory at three different time scales. Each layer lives in a different place — and forgets at a different speed.

⚡

In-turn (live)

The active turn’s tool outputs, model tokens, current input queue. Disappears when the turn ends if not promoted.

📜

In-session (rollout)

Everything in the current conversation, on disk as rollout.jsonl. Survives crashes. Resumable via codex resume.

🗂️

Across sessions

Long-term notes the agent writes to ~/.codex/memories/ and skills configured in user / project scope. Survive across all sessions.

The middle layer — the rollout — is where most of the interesting machinery lives. The rest of this module is about that one file.

What the logbook actually looks like

Every conversation is one file on your disk. Each line is one event. Open it in any text editor — it is just JSONL.

FILE — ~/.codex/threads/…/rollout.jsonl 
# ~/.codex/threads/2026-05-27/{thread_id}/rollout.jsonl

{"type":"TurnStarted","timestamp":...}
{"type":"UserMessage","text":"fix the failing test"}
{"type":"AgentMessage","text":"Looking at the test file first..."}
{"type":"FunctionCall","tool":"shell","args":{"cmd":"pnpm test"}}
{"type":"FunctionResult","output":"error in src/parser.ts line 47..."}
{"type":"FunctionCall","tool":"apply_patch","args":{...}}
{"type":"FunctionResult","output":"patch applied"}
{"type":"AgentMessage","text":"Done. The typo was on line 47."}
{"type":"TurnEnded","timestamp":...}
PLAIN ENGLISH

Each line is one event — one thing that happened, in the order it happened.

User messages, agent messages, tool calls, tool results — all written down as they occur, no editing afterward.

The file is append-only. New events go on the bottom. Old events are never rewritten.

To re-create the conversation in memory, you just read the file from top to bottom and replay every event. That is what codex resume does.

💡
Aha! moment: append-only is a superpower

Because the log only grows — never edits or deletes — you can fork it (start a side thread from any point), replay it (debug exactly what happened), or rebuild from it (recover from a crash). Engineers call this event sourcing — and it’s the same trick git uses for your commits.

A Session, in code

The rollout is the on-disk shape. The Session is the in-memory shape. Here is what one conversation looks like to Codex while it is running.

CODE — core/src/session/session.rs 
pub(crate) struct Session {
    pub(crate) conversation_id: ThreadId,
    pub(crate) installation_id: String,
    pub(super) tx_event: Sender<Event>,
    pub(super) agent_status: watch::Sender<AgentStatus>,
    pub(super) state: Mutex<SessionState>,
    pub(super) conversation: Arc<RealtimeConversationManager>,
    pub(crate) active_turn: Mutex<Option<ActiveTurn>>,
    pub(crate) input_queue: InputQueue,
    pub(super) goal_runtime: GoalRuntimeState,
    // ...
}
PLAIN ENGLISH

conversation_id: a unique ID for this chat — like a file name. Engineers call it a ThreadId.

tx_event: the wire that carries events out to the UI. Every time something happens, an event goes down this wire and the TUI sees it.

state: the in-memory bookkeeping, wrapped in a Mutex so two pieces of code never trample each other.

active_turn: the in-progress turn if there is one, otherwise nothing. Most of the time this is empty — Codex is waiting for you.

input_queue: the holding pen where your keystrokes wait if you type while the model is still thinking.

Together: this is everything one conversation needs to be itself.

The life of one session

From codex to codex resume, five things have to happen. Each step is just a page in the logbook.

1
Session starts

A ThreadId is minted; a new rollout.jsonl file is opened on disk.

2
Each turn appends events

User message, model output, function calls, function results — every event is written to disk as it happens.

3
Context window fills up

A token counter detects the prompt is getting close to the model’s context window limit.

4
Compaction kicks in

A special turn runs: “summarize the old history.” The summary replaces those old turns in the prompt; the rollout file still has the raw events.

5
Resume later

codex resume reads the rollout file, rebuilds the session in memory, and you continue.

Compaction — squeezing the past

The model’s context window has a ceiling. When the conversation pushes against it, Codex runs a special turn whose job is to summarize the older pages of the logbook. Same machinery as any other turn — but the answer becomes the new history.

CODE — core/src/compact.rs 
pub(crate) async fn run_inline_auto_compact_task(
    sess: Arc<Session>,
    turn_context: Arc<TurnContext>,
    initial_context_injection: InitialContextInjection,
    reason: CompactionReason,
    phase: CompactionPhase,
) -> CodexResult<()> {
    let prompt = turn_context.compact_prompt().to_string();
    let input = vec![UserInput::Text {
        text: prompt,
        text_elements: Vec::new(),
    }];

    run_compact_task_inner(
        sess,
        turn_context,
        input,
        initial_context_injection,
        CompactionTrigger::Auto,
        reason,
        phase,
    )
    .await?;
    Ok(())
}
PLAIN ENGLISH

Build a special prompt that essentially says: “summarize everything above this point.”

Wrap that prompt as if you had typed it — same shape as a normal user message.

Run it as a turn, using the exact same loop as any other turn. There is no separate “summarizer” engine — just the model, given a different job for one round.

The model’s answer is the summary. That summary gets injected into the in-memory history in place of the older turns. The original logbook on disk is untouched.

Then the regular conversation resumes — with room to breathe again.

💡
Aha! moment: the disk log and the model’s view are two different things

The compaction step rewrites the model’s view of history, but the rollout file always has the complete history. The model sees a smaller view that fits its context window. Compaction is what bridges the gap. This is why you can codex resume a three-hour session and find perfect recall — even when the model itself only remembers a summary of most of it.
